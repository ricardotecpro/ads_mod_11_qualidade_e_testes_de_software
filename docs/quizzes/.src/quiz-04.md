# Quiz 04 - Documentação e Controle de Testes 📄

1. Qual artefato descreve o escopo, a estratégia e o cronograma de testes?
   - [ ] Caso de Teste.
   - [x] Plano de Teste.
   - [ ] Relatório de Bug.
   - [ ] Manual do Usuário.
   > **Explicação**: O Plano de Teste é o documento mestre de organização.

2. Um "Caso de Teste" deve conter obrigatoriamente:
   - [ ] O CPF do testador.
   - [ ] O código-fonte completo.
   - [x] Passos de execução e Resultado Esperado.
   - [ ] Apenas o título.
   > **Explicação**: Passos e resultados esperados são a essência de um caso de teste.

3. Qual a diferença entre Prioridade e Severidade?
   - [ ] São a mesma coisa.
   - [x] Severidade olha o impacto técnico; Prioridade olha a urgência de negócio.
   - [ ] Prioridade olha o impacto técnico; Severidade olha o negócio.
   - [ ] Severidade é para bugs visuais apenas.
   > **Explicação**: Um bug pode ser severo (trava o banco), mas de baixa prioridade (em uma área pouco usada).

4. O termo "Rastreabilidade de Teste" refere-se a:
   - [ ] Saber onde o testador está fisicamente.
   - [x] Ligar requisitos aos casos de teste e aos bugs encontrados.
   - [ ] Localizar o IP do servidor de testes.
   - [ ] Contar o número de horas trabalhadas.
   > **Explicação**: Rastreabilidade garante que todos os requisitos foram testados.

5. O que é um "Cenário de Teste"?
   - [ ] O mesmo que Caso de Teste.
   - [x] Uma descrição de alto nível do que será testado (ex: Validar Login).
   - [ ] Uma foto da tela com bug.
   - [ ] O ambiente onde o teste roda.
   > **Explicação**: O cenário é "o que" testar; o caso de teste é "como" testar detalhadamente.

6. No Report de Bug, os "Passos para Reproduzir" servem para:
   - [ ] Encher o documento de texto.
   - [x] Permitir que o desenvolvedor veja o erro acontecer em sua própria máquina.
   - [ ] Provar que o QA trabalhou muito.
   - [ ] Substituir o código do desenvolvedor.
   > **Explicação**: Reproduzir o bug é o primeiro passo para corrigi-lo.

7. Qual resultado deve ser registrado em um Plano de Testes quando o sistema não atende ao esperado?
   - [ ] Bloqueado.
   - [x] Falha (Fail).
   - [ ] Sucesso (Pass).
   - [ ] Ignorar.
   > **Explicação**: Quando o comportamento real difere do esperado, temos uma falha.

8. O que são "Pré-condições" em um caso de teste?
   - [ ] O clima no dia do teste.
   - [ ] A configuração do salário do QA.
   - [x] Estados necessários do sistema antes do teste começar (ex: Usuário Logado).
   - [ ] O resultado final esperado.
   > **Explicação**: Pré-condições garantem que o teste comece do ponto correto.

9. Métricas de teste (ex: % de execução) são usadas para:
   - [ ] Comparar o desempenho de desenvolvedores entre si.
   - [x] Fornecer visibilidade sobre o progresso e a qualidade do produto.
   - [ ] Apenas para o cliente ver gráficos bonitos.
   - [ ] Reduzir o salário da equipe.
   > **Explicação**: Métricas apoiam a tomada de decisão baseada em fatos.

10. Se um "Caso de Teste" falha, qual a próxima ação imediata do QA?
    - [ ] Apagar o teste.
    - [x] Abrir um Relatório de Bug vinculando ao caso de teste.
    - [ ] Tentar rodar o teste 100 vezes até passar.
    - [ ] Mandar o desenvolvedor recomeçar o projeto.
    > **Explicação**: A falha gera um report de defeito para correção.
